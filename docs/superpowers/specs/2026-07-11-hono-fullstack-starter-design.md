# Hono Full-Stack Starter Design

## Goal

Turn the starter into a small full-stack Node.js monorepo example while preserving the renderer showcase. Replace the unused, domain-specific `@cheshirecode/brief-schema` package with a reusable Hono foundation package, and evolve `app-react` from a client-only Vite app into the sole deployable Hono + React SSR application.

## Scope

- Remove `@cheshirecode/brief-schema` and every package-specific documentation, dogfood, changeset, and lockfile reference.
- Add publishable `@cheshirecode/hono-base`.
- Evolve private `app-react` into an isomorphic Hono + React application on Node.js.
- Preserve `app-react` as a renderer demo mounted by `renderer-showcase`.
- Include the Hono package and SSR React app in repositories generated by `@cheshirecode/create-moon-pnpm-monorepo`.
- Keep the root framework-neutral and retain pnpm, Moon, Changesets, Vite, Vitest, oxlint, Node 24, and TypeScript 7 conventions.

Deno runtime integration is intentionally out of scope. Hono route code must remain Web-Standard so a future Deno adapter can use the same app factory, but the current starter has one package manager, one lockfile, one CI runtime, and one production server runtime.

## Architecture

### `@cheshirecode/hono-base`

A publishable package under `packages/hono-base` exports an app factory rather than starting a process:

```ts
createBaseApp({ version, serviceName? })
```

The returned Hono instance contains:

- `GET /healthz` returning HTTP 200 and `{ "status": "ok" }`.
- `GET /version` returning HTTP 200 and stable service/version metadata supplied by the caller.
- JSON 404 responses for unmatched API routes.
- A centralized unexpected-error response that returns generic JSON without error internals.

The package uses only Hono and Web APIs. It must not import `node:*`, `@hono/node-server`, React, Vite, filesystem APIs, environment variables, or package metadata at runtime. This keeps it directly testable with `app.request()` and portable to another Hono runtime adapter.

### `app-react`

`app-react` remains private and becomes the sole deployable server. It depends on `@cheshirecode/hono-base`, calls `createBaseApp()`, and extends the returned app with React document routes.

Runtime-specific code is split by responsibility:

- A shared React component tree contains deterministic application UI.
- A browser entry hydrates server-rendered markup.
- The existing microfrontend entry mounts the same component tree with `createRoot`, preserving the renderer showcase contract.
- A server app composes base API routes with SSR routes.
- A Node entry owns `@hono/node-server`, port configuration, startup, and graceful `SIGINT`/`SIGTERM` shutdown.

The server-rendered document includes the Vite-built client entry and styles through production manifest data or an equivalent deterministic asset mapping. Development uses Vite middleware or a narrowly scoped Vite development integration inside `app-react`; root tooling stays framework-neutral.

SSR uses React's stable server renderer. Streaming, Suspense orchestration, and Hono JSX are excluded unless required by the selected stable Vite integration: they add complexity without improving this static demonstration and Hono's JSX streaming APIs remain a separate rendering model from React hydration.

## Data Flow

1. Node receives a request through `@hono/node-server`.
2. Hono dispatches `/healthz` and `/version` through `@cheshirecode/hono-base`.
3. `app-react` handles its document route by rendering the shared React tree on the server.
4. The response embeds only deterministic, safely serialized bootstrap data needed to reproduce the same first render.
5. The browser client hydrates the existing server markup rather than replacing it.
6. When loaded inside `renderer-showcase`, the package-local microfrontend adapter mounts the same tree into the host-provided element without involving the SSR server.

## Response Contracts

`GET /healthz`:

```json
{
  "status": "ok"
}
```

`GET /version`:

```json
{
  "name": "app-react",
  "version": "0.0.0"
}
```

The version value comes from explicit composition-time configuration, not runtime filesystem discovery. Response bodies and content types are asserted in tests so generated repositories inherit a stable example contract.

## Error Handling and Security

- API 404 and 500 responses are JSON and do not expose stack traces, filesystem paths, request headers, or environment variables.
- SSR failures use Hono's centralized error path and return a minimal HTML 500 response.
- Server-side logs may record the original error, but tests must not depend on log text.
- Bootstrap data is serialized through a safe JSON-to-script mechanism that prevents closing-tag injection.
- Static assets resolve from paths derived from `import.meta.url`, not the caller's working directory.
- The Node server closes cleanly on `SIGINT` and `SIGTERM`.

## Package and Release Model

`@cheshirecode/hono-base` is publishable and follows repository metadata, README, license, build, type declaration, coverage, pack, dogfood, and Changesets rules. Its dogfood consumer imports the packed tarball and invokes routes through `app.request()`.

`app-react` remains private and is excluded from Changesets. It depends on `@cheshirecode/hono-base` with the workspace protocol and declares Moon dependencies so package builds occur in dependency order.

Removing `brief-schema` removes its AJV dependencies, self-tests, dogfood assertions, package documentation, README graph node, package changeset, and remaining package-specific changeset references.

## Generator

Generated repositories include:

- The publishable `@cheshirecode/hono-base` package.
- The private Hono + React SSR `app-react` application.
- Dependency-aware Moon metadata.
- Root scripts and CI behavior sufficient to install, lint, typecheck, build, test, pack, and dogfood the generated repository from a clean checkout.

Generator tests assert both projects and their essential files, dependency relationship, TypeScript 7 defaults, and full-stack verification commands. Dogfood runs generated CI from both API- and CLI-generated repositories.

## Renderer Showcase Compatibility

The renderer registry continues to contain exactly the six existing renderer applications. `app-react` keeps its current package-local microfrontend export and `RendererDemoContract` behavior. The SSR server is an additional execution mode, not a replacement for the host-shell mount contract.

The showcase does not proxy to or embed the Node server. Its React demo imports the client-safe microfrontend entry only, preventing Node modules from entering the browser bundle.

## Testing and Verification

Tests are layered:

- `hono-base`: route status, body, content type, 404, and generic 500 behavior through `app.request()`.
- `app-react` server: SSR response status, HTML shell, escaped bootstrap data, API inheritance, and failure fallback.
- `app-react` browser: hydration without mismatch warnings and continued interactive behavior.
- `app-react` microfrontend: mount/unmount contract remains valid.
- `renderer-showcase`: registry and build smoke checks continue to pass.
- Generator: generated file inventory, workspace dependency, clean install, CI, pack, and package dogfood.

Focused development uses Moon package targets. Before handoff, run:

```sh
pnpm moon run hono-base:lint
pnpm moon run hono-base:typecheck
pnpm moon run hono-base:build
pnpm moon run hono-base:test
pnpm moon run app-react:lint
pnpm moon run app-react:typecheck
pnpm moon run app-react:build
pnpm moon run app-react:test
scripts/check.sh package-drift
scripts/check.sh renderer-showcase
scripts/check.sh pack
scripts/check.sh dogfood packages
scripts/check.sh changeset-check
scripts/check.sh ci
```

Inspect `.artifacts/dogfood/report.json` after dogfood. Broaden to generated-repository dogfood and Docker verification if server startup, generated setup, or container behavior changes require it.

## Success Criteria

- No `brief-schema` package or package-specific reference remains.
- `@cheshirecode/hono-base` is reusable, publishable, packed, and dogfooded.
- `app-react` serves `/healthz`, `/version`, and server-rendered React on Node.js.
- The browser hydrates SSR output and the renderer showcase still mounts `app-react` as a microfrontend.
- Generated repositories contain and verify the same full-stack shape.
- Package drift, renderer showcase, package-facing checks, changeset checks, and CI pass without adding TypeScript 6 exceptions or root framework assumptions.
